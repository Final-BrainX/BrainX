# BrainX Secure Coding Checklist

- 작성일: 2026-07-09
- 대상: `brainX_back/`, `brainx-next/`, `contracts-v2/`
- 기준: `contracts-v2/brainx-openapi.ssot.yaml`, `contracts-v2/brainx-asyncapi.ssot.yaml`, `README.md`

## 목적

이 문서는 BrainX 저장소 전반에 시큐어 코딩을 적용할 때 사용하는 실무 점검표다.
프론트엔드, 백엔드, 서비스 간 호출, 문서화와 테스트까지 같은 기준으로 점검하는 것을 목표로 한다.

## 현재 기준 갭 요약

현재 코드베이스를 빠르게 대조했을 때 확인된 상태는 다음과 같다.

- 입력 검증: Spring 컨트롤러의 `@Valid`, `@Validated`, 전역 예외 처리기는 이미 일부 적용되어 있다.
- 인증/권한: 서비스별 `SecurityConfig`와 JWT 필터가 존재하지만, 일부 공개 범위와 CORS 허용 범위는 더 엄격하게 줄일 필요가 있다.
- SQL 인젝션: 전반적으로 JPA/Repository 중심이라 방향은 좋지만 `JdbcTemplate`, native query, 동적 정렬/필터 구간은 별도 감사가 필요하다.
- XSS: `dangerouslySetInnerHTML`, `innerHTML`, HTML 후처리 로직이 존재해 우선 점검 대상이다.
- 예외/로그: 전역 예외 처리 패턴은 존재하지만, 클라이언트 응답 메시지와 서버 상세 로그의 분리가 계속 유지되는지 검증이 필요하다.

우선순위가 높은 현재 점검 후보:

- `brainx-next/components/public/share-screen.tsx`
- `brainx-next/components/home-screen.tsx`
- `brainx-next/components/notes/NoteEditor.tsx`
- `brainx-next/lib/notes/exportNoteContent.ts`
- `brainX_back/Workspace-Service/src/main/java/com/brainx/workspace/config/SecurityConfig.java`

## 점검 원칙

1. 프론트 검증은 UX용이고, 백엔드 검증은 보안용이다.
2. 사용자 입력, 외부 연동 데이터, AI 생성 결과, 파일 메타데이터는 모두 신뢰하지 않는다.
3. SSOT에 없는 입력값, 응답 형태, 권한 흐름은 구현하지 않는다.
4. 공개 API, 로그인 사용자 API, 내부 서비스 API는 보안 수준을 다르게 적용한다.
5. 상세 원인은 로그에 남기고, 사용자 응답에는 일반화된 메시지만 노출한다.

## 저장소 전체 체크리스트

### 1. 입력 데이터 검증

- [ ] 모든 REST 요청 DTO가 `@Valid` 또는 `@Validated` 경로에 들어오는지 확인한다.
- [ ] 문자열 입력에 길이 제한이 있는지 확인한다.
- [ ] 이메일, URL, UUID, enum 값은 형식 검증이 있는지 확인한다.
- [ ] `title`, `nickname`, `tag`, `folderName`, `workspaceName` 같은 사용자 입력 필드에 허용 범위가 정의돼 있는지 확인한다.
- [ ] nullable 필드가 실제 비즈니스 규칙과 일치하는지 확인한다.
- [ ] 숫자 입력은 최소/최대 범위를 제한한다.
- [ ] 배열 입력은 최대 개수 제한이 있는지 확인한다.
- [ ] 정렬 기준, 필터 키, 화면 모드 값처럼 제어 목적의 입력은 화이트리스트 기반인지 확인한다.
- [ ] 프론트에서 막더라도 백엔드가 동일한 검증을 다시 수행하는지 확인한다.
- [ ] 검증 규칙이 SSOT 스키마와 충돌하지 않는지 확인한다.

### 2. 인증과 인가

- [ ] 인증이 필요한 엔드포인트는 `permitAll()`로 열려 있지 않은지 확인한다.
- [ ] 공개 API, 사용자 API, 관리자 API, 내부 서비스 API가 분리되어 있는지 확인한다.
- [ ] 리소스 소유권 검증이 있는지 확인한다.
- [ ] `noteId`, `folderId`, `documentGroupId`, `shareId` 조회 시 현재 사용자 또는 서비스 권한과 연결 검증이 있는지 확인한다.
- [ ] Guest 경로는 명시적으로만 허용되는지 확인한다.
- [ ] 개발용 fallback auth가 운영에서 기본 비활성인지 확인한다.
- [ ] 내부 `/internal/v1/**` API는 서비스 토큰 또는 서비스 전용 role로만 제한되는지 확인한다.
- [ ] 관리자 API는 일반 사용자 role과 분리되어 있는지 확인한다.
- [ ] 권한 실패 시 401/403 응답이 SSOT와 일치하는지 확인한다.

### 3. SQL 인젝션 방지

- [ ] SQL 문자열 연결로 where/order/limit를 조립하지 않는지 확인한다.
- [ ] `PreparedStatement`, JPA parameter binding, `JdbcTemplate` 바인딩만 사용하는지 확인한다.
- [ ] native query 사용 시 입력값이 바인딩되는지 확인한다.
- [ ] 정렬 컬럼명, 테이블명, 검색 대상 컬럼처럼 바인딩이 불가능한 값은 화이트리스트로 매핑하는지 확인한다.
- [ ] 검색 기능에서 `%keyword%` 조합 시에도 쿼리 본문이 아니라 파라미터 값으로만 넣는지 확인한다.
- [ ] 마이그레이션/관리성 코드가 런타임 사용자 입력을 직접 받지 않는지 확인한다.

### 4. XSS 방지

- [ ] `dangerouslySetInnerHTML` 사용 지점을 전수 파악한다.
- [ ] `innerHTML` 직접 대입 지점을 전수 파악한다.
- [ ] Markdown -> HTML 변환 결과를 그대로 렌더링하지 않는지 확인한다.
- [ ] 공유 페이지, AI 응답, import 결과 HTML은 sanitizer를 거치는지 확인한다.
- [ ] 정규식 기반 태그 제거만으로 안전하다고 가정하지 않는지 확인한다.
- [ ] 링크 렌더링 시 `javascript:` 같은 위험 스킴을 차단하는지 확인한다.
- [ ] iframe 사용 시 `sandbox` 속성이 필요한 최소 범위만 열려 있는지 확인한다.
- [ ] 사용자 텍스트는 가능한 한 text node로 렌더링하고, HTML 허용은 예외 경로로만 관리하는지 확인한다.
- [ ] CSP 적용 여부와 inline script/style 의존도를 검토한다.

### 5. 파일 업로드와 다운로드

- [ ] 업로드 파일의 확장자와 MIME type을 모두 검증하는지 확인한다.
- [ ] 파일명은 저장 전에 정규화하는지 확인한다.
- [ ] 경로 순회(`../`)가 불가능한지 확인한다.
- [ ] 다운로드 응답에서 의도하지 않은 로컬 경로나 내부 스토리지 키를 노출하지 않는지 확인한다.
- [ ] 이미지/PDF/HTML 미리보기는 신뢰되지 않는 콘텐츠로 취급하는지 확인한다.
- [ ] 압축 파일 import 시 zip slip 방어가 있는지 확인한다.
- [ ] 최대 업로드 크기와 타임아웃이 정의돼 있는지 확인한다.

### 6. 예외 처리와 에러 응답

- [ ] 전역 예외 처리기가 일관된 응답 형식을 사용하는지 확인한다.
- [ ] 사용자 응답에 stack trace, SQL 문장, 내부 클래스명, 토큰, 비밀값이 노출되지 않는지 확인한다.
- [ ] `ApiException` 계열 메시지가 내부 구현 세부사항을 직접 담지 않는지 확인한다.
- [ ] 검증 오류, 인증 오류, 권한 오류, 비즈니스 오류, 시스템 오류가 구분되는지 확인한다.
- [ ] `traceId` 또는 `requestId`가 항상 남는지 확인한다.

### 7. 로그와 민감정보 처리

- [ ] access token, refresh token, API key, 비밀번호, 인증코드 원문을 로그에 남기지 않는지 확인한다.
- [ ] 이메일, 전화번호, 결제정보, IP 주소는 필요한 범위에서만 기록하는지 확인한다.
- [ ] 예외 로그에 요청 본문 전체를 그대로 덤프하지 않는지 확인한다.
- [ ] 운영 로그와 개발 디버그 로그의 레벨 정책이 분리되어 있는지 확인한다.
- [ ] 민감한 설정값은 환경변수 또는 secret store에서만 읽는지 확인한다.

### 8. CORS, 헤더, 브라우저 보안

- [ ] `allowCredentials=true`일 때 허용 origin이 와일드카드처럼 넓게 열려 있지 않은지 확인한다.
- [ ] 운영/개발 origin 목록이 분리되어 있는지 확인한다.
- [ ] `Authorization` 외 불필요한 exposed header가 없는지 확인한다.
- [ ] `X-Frame-Options`, `Content-Security-Policy`, `Referrer-Policy`, `X-Content-Type-Options` 적용 여부를 검토한다.
- [ ] 외부 링크는 `rel=\"noreferrer noopener\"` 등 안전 속성을 사용하는지 확인한다.

### 9. 세션, 토큰, 쿠키

- [ ] JWT 검증 시 만료, 서명, issuer/audience, role 파싱이 일관되는지 확인한다.
- [ ] 타이밍 공격 방어가 필요한 비교는 상수 시간 비교를 사용하는지 확인한다.
- [ ] refresh token 저장/폐기 정책이 문서화돼 있는지 확인한다.
- [ ] 브라우저 저장소에 장기 민감정보를 불필요하게 저장하지 않는지 확인한다.
- [ ] 데스크톱 브리지 저장소도 동일한 민감정보 정책을 따르는지 확인한다.

### 10. 서비스 간 호출과 비동기 이벤트

- [ ] `/internal/v1/**` 호출은 서비스 인증 없이 열려 있지 않은지 확인한다.
- [ ] 서비스 간 요청에도 입력 검증이 적용되는지 확인한다.
- [ ] AsyncAPI 이벤트 payload에 민감정보 원문이 불필요하게 포함되지 않는지 확인한다.
- [ ] 이벤트 consumer는 eventId 기준 멱등성을 지키는지 확인한다.
- [ ] 로그/이벤트에 외부 provider secret이나 raw prompt 전체가 과도하게 남지 않는지 확인한다.

### 11. AI 및 생성형 기능

- [ ] AI 입력 프롬프트에 민감정보가 과도하게 포함되지 않는지 확인한다.
- [ ] AI가 생성한 HTML/Markdown/링크를 그대로 신뢰하지 않는지 확인한다.
- [ ] 외부 검색/도구 호출 결과도 사용자 입력과 동일하게 검증하는지 확인한다.
- [ ] 모델 응답 실패 메시지에 provider 내부 오류를 그대로 노출하지 않는지 확인한다.
- [ ] 토큰 사용량 기록 이벤트에 사용자 비밀값이 섞이지 않는지 확인한다.

### 12. 프론트엔드 상태와 UX 보안

- [ ] 권한이 없는 기능은 버튼만 숨기지 말고 API에서도 차단되는지 확인한다.
- [ ] URL 파라미터만 바꿔 다른 사용자 데이터에 접근할 수 없는지 확인한다.
- [ ] 공유 링크 만료/비활성/권한없음 상태가 구분되어 노출되는지 확인한다.
- [ ] 에러 토스트나 화면 문구에 내부 시스템 세부정보가 노출되지 않는지 확인한다.

### 13. 테스트

- [ ] 입력 검증 실패 테스트가 존재하는지 확인한다.
- [ ] 인증 없음, 권한 없음, 타 사용자 리소스 접근 거부 테스트가 존재하는지 확인한다.
- [ ] XSS payload 회귀 테스트가 존재하는지 확인한다.
- [ ] SQL 인젝션성 입력을 넣는 회귀 테스트가 존재하는지 확인한다.
- [ ] 공유 링크와 guest 경로에 대한 보안 테스트가 존재하는지 확인한다.
- [ ] 개발용 auth fallback이 운영 기본값에서 차단되는 테스트가 존재하는지 확인한다.

### 14. 문서와 SSOT 동기화

- [ ] OpenAPI `security`, `401`, `403`, validation schema가 실제 구현과 일치하는지 확인한다.
- [ ] AsyncAPI payload에 `documentGroupId` 같은 경계 정보가 빠지지 않는지 확인한다.
- [ ] README와 운영 문서에 보안 관련 기본 정책이 반영되어 있는지 확인한다.
- [ ] 새 보안 규칙이 생기면 체크리스트와 SSOT 영향도를 함께 검토한다.

## 권장 실행 순서

1. XSS 렌더링 지점 점검
2. `SecurityConfig`와 공개 엔드포인트 점검
3. DTO 검증 규칙 보강
4. SQL/native query 감사
5. 로그/예외 응답 정리
6. 보안 회귀 테스트 추가
7. README와 SSOT 영향 반영

## PR 리뷰용 짧은 체크리스트

PR 단위로는 아래만 먼저 확인해도 된다.

- [ ] 이 변경이 새 입력값을 받는가
- [ ] 그 입력값은 백엔드에서 검증되는가
- [ ] 권한 없는 사용자가 호출해도 막히는가
- [ ] HTML 렌더링/XSS 위험은 없는가
- [ ] 로그와 에러 응답에 민감정보가 남지 않는가
- [ ] SSOT 문서와 구현이 어긋나지 않는가

## 운영 메모

- 이 문서는 “보안 정책 선언”이 아니라 “실제 구현 점검표”다.
- 점검 결과는 HIGH, MEDIUM, LOW로 분류해 별도 이슈로 관리하는 것을 권장한다.
- BrainX 현재 단계에서는 XSS, 권한 범위, guest/dev fallback, 내부 API 보호를 최우선으로 본다.
