# BrainX 최종 발표·실무 기술면접 예상 질문과 모범 답변

> **기준일:** 2026-07-10
> **검증 범위:** `contracts-v2`의 OpenAPI·AsyncAPI SSOT, `brainX_back`의 9개 서비스, 웹·Electron·Chrome Extension 채널, 인프라 문서를 대조했다. 과거 계획성 README보다 현재 계약·코드·설정을 우선한다.

## 사용 방법

- 각 **모범 답변**은 30~60초 분량이다. 결론을 먼저 말하고, 사용자 가치·서비스 경계·현재 한계를 순서대로 설명한다.
- **꼬리질문 대비**는 더 깊게 물을 때만 사용한다. 구현되지 않은 내용을 구현 사실처럼 확장하지 않는다.
- 개인 기여도는 마지막 템플릿에 본인이 실제로 맡고 검증한 범위만 넣는다.

## 한 페이지 요약

### BrainX가 해결하려는 사용자 문제

사용자는 메모를 많이 남겨도 키워드를 잊으면 다시 찾지 못하고, 웹·파일·외부 도구에서 들어온 자료가 흩어져 있으면 지식으로 연결하기 어렵다. BrainX는 노트를 **신뢰할 수 있는 원본으로 보관**하고, 그 위에 검색·RAG 대화·작성 보조·그래프·정리 제안을 올려 “축적한 지식을 다시 쓰는 경험”을 만든다.

### 차별점

BrainX의 핵심은 범용 AI 답변이 아니라 **사용자 소유 Workspace 지식에 근거한 활용**이다. 원본 노트는 Workspace가 소유하고, Intelligence는 이를 복제해 검색·분석하며, AI가 제안한 변경은 사용자가 검토·승인한 뒤에만 원본으로 반영한다. 수집·결제·관리·MCP를 별도 도메인으로 나누어 개인 지식 관리의 전체 여정을 연결한다.

### 서비스 책임 분리

| 도메인/서비스 | 사용자·업무 책임 | 기준 데이터 또는 경계 |
| --- | --- | --- |
| User-Service | 가입·로그인·OAuth·계정·동의·알림·문의 | 사용자 신원과 세션 |
| Workspace-Service | 노트·폴더·태그·링크·공유·그래프 | 원본 지식의 source of truth |
| Ingestion-Service | 파일·Notion·Obsidian·웹 캡처 유입, export/publish | 외부 콘텐츠를 Workspace 명령으로 반영 |
| Intelligence-Service | 시맨틱 검색·RAG·작성 보조·추천·분석 | AI용 projection, Qdrant index, 대화·분석 결과 |
| Commerce-Service | 플랜·결제·구독·entitlement·token usage ledger | 사용 권한과 비용/청구 원장 |
| MCP-Service | 외부 AI agent의 사용자 범위 접근 | scope가 있는 MCP API client와 도구 gateway |
| Admin-Service | 운영자 계정·고객·결제·토큰·지원·모니터링 | 운영용 조회/명령과 snapshot |
| Gateway/Discovery | 외부 진입 인증·라우팅 / 서비스 발견 | 비즈니스 원장은 소유하지 않음 |

### 대표 사용자 흐름

가입 또는 게스트 체험 → Workspace에서 노트 작성·가져오기·웹 캡처 → Workspace 이벤트와 snapshot으로 Intelligence 색인 → 검색·RAG·작성 보조·그래프 활용 → token usage를 Commerce에 기록 → Admin에서 운영 지표와 지원·결제 이슈를 처리한다. 외부 AI agent는 MCP scope 안에서 같은 사용자 지식에 접근한다.

### 발표에서 피해야 할 표현

| 말하면 안 되는 주장 | 현재 사실 |

## 1. 기획·도메인·사용자 가치

### 1. BrainX는 어떤 제품이고, 누구의 문제를 해결하나요?

**상태:** ✅ 구현 방향
**모범 답변:** BrainX는 개인이 쌓아 둔 노트와 외부 자료를 다시 찾고, 연결하고, 활용하도록 돕는 지식 운영 플랫폼입니다. 핵심 사용자는 기록은 하지만 과거 맥락을 찾기 어렵거나 자료가 여러 채널에 흩어진 사람입니다. 단순 저장에 그치지 않고 Workspace 원본을 기반으로 검색·AI 대화·작성 보조·그래프를 연결해, 기록량이 늘수록 재사용 가치도 커지게 설계했습니다.
**근거:** [루트 제품 개요], [Workspace 아키텍처], [Intelligence 도메인 흐름]

### 2. 일반 노트 앱이나 범용 생성형 AI와 어떤 점이 다른가요?

**상태:** ✅ 구현됨
**모범 답변:** 범용 생성형 AI는 대화 한 번의 결과에 집중하는 반면, BrainX는 사용자의 Workspace를 검색 근거로 삼는다는 점이 다릅니다. 원본 노트는 Workspace가 보존하고 Intelligence가 RAG와 시맨틱 검색으로 활용하며, 웹 캡처·파일 import·MCP도 결국 그 원본 지식으로 흘려보냅니다. 따라서 “AI가 똑똑하다”보다 “내가 축적한 지식을 출처와 함께 다시 사용할 수 있다”가 발표의 중심입니다.
**꼬리질문 대비:** RAG가 환각을 완전히 없앤다고 주장하지 않고, 근거 제시와 품질 측정으로 위험을 줄인다고 설명합니다.
**근거:** [Workspace 아키텍처], [RAG 채팅], [MCP 노트 도구]

### 3. 왜 도메인을 여러 서비스로 나눴나요?

**상태:** ✅ 구현된 경계
**모범 답변:** BrainX의 핵심 업무는 원본 지식, 사용자 신원, AI 처리, 결제 권한, 외부 유입, 운영 관리로 나뉘고 각각 변경 주기와 장애 영향이 다릅니다. 그래서 Workspace는 원본 지식을, Commerce는 entitlement와 결제 원장을, Intelligence는 AI용 파생 데이터를 맡도록 분리했습니다. 이 경계는 한 서비스의 기술 선택이나 장애가 다른 도메인의 기준 데이터를 직접 훼손하지 않도록 하는 장치입니다.
**근거:** [OpenAPI SSOT], [서비스 카탈로그], [AsyncAPI 가이드]

### 4. BrainX의 진짜 기준 데이터는 무엇인가요?

**상태:** ✅ 구현됨
**모범 답변:** 노트·폴더·태그·링크·공유 같은 사용자 지식의 기준 데이터는 Workspace-Service입니다. Intelligence의 노트 projection, vector index, 요약 cache는 빠른 검색과 AI 기능을 위한 복제본이므로 eventual consistency가 있을 수 있습니다. 그래서 AI 결과가 원본을 직접 바꾸지 않고, 원본 변경은 Workspace command로만 처리하는 원칙을 유지합니다.
**근거:** [Workspace 아키텍처], [Intelligence 데이터 소유권], [교차 서비스 경계]

### 5. 사용자의 핵심 여정은 어떻게 이어지나요?

**상태:** ✅ 구현됨
**모범 답변:** 사용자는 가입 후 기본 Workspace에서 노트를 작성하거나, 파일·Notion·Obsidian·웹 캡처로 자료를 들여옵니다. 원본 변경은 Workspace에 저장되고 이벤트와 snapshot을 통해 Intelligence의 검색 projection으로 반영됩니다. 이후 사용자는 검색·RAG 대화·인라인 작성 보조·링크/폴더 제안·그래프로 지식을 활용하고, AI 사용량은 Commerce 기록을 거쳐 운영자가 Admin에서 확인할 수 있습니다.
**근거:** [AsyncAPI 가이드], [Ingestion controller], [AI 기능 카탈로그], [Commerce 사용량]

### 6. 게스트 체험을 제공하는 이유와 한도는 무엇인가요?

**상태:** ✅ 구현됨
**모범 답변:** 가입 전에도 노트 작성과 일부 AI 경험을 제공하면 제품의 핵심 가치를 먼저 확인하게 할 수 있습니다. Gateway는 로그인 토큰이 없는 허용 경로에 `gst_` 형식의 guest ID cookie를 발급하고, Workspace는 게스트 draft를 Redis에 보관하며, Commerce entitlement는 게스트 합산 10회 한도를 판정합니다. 다만 게스트는 영구 Workspace 원본을 보유하지 않으므로 가입 후 draft를 claim하는 전환 흐름이 필요합니다.
**근거:** [Gateway 게스트 인증], [Workspace 정책], [Commerce entitlement]

### 7. 웹·데스크톱·브라우저 확장은 각각 어떤 역할을 하나요?

**상태:** ✅ 채널 구현, ⚠️ 데스크톱 local-first는 진행 중
**모범 답변:** 웹은 노트, 채팅, 그래프, import, 결제, 설정을 제공하는 주 사용자 채널입니다. Electron은 같은 웹 경험을 설치형 데스크톱으로 제공하면서 OAuth·결제 popup과 파일 dialog를 desktop bridge로 연결하고, Chrome Extension은 웹 페이지와 선택 텍스트를 캡처해 노트로 저장합니다. Electron의 vault 폴더와 metadata 기반은 있지만 실제 노트/asset 저장을 local vault로 완전히 전환하고 offline-first sync를 만드는 일은 남아 있습니다.
**근거:** [웹 라우트], [Electron README], [Chrome Extension manifest]

### 8. 제품 성공을 무엇으로 측정하겠나요?

**상태:** ⚠️ 측정 기반은 있으나 KPI 목표는 후속 정의
**모범 답변:** BrainX에서는 단순 가입자 수보다 “사용자가 자기 지식을 다시 찾고 활용했는가”가 더 중요합니다. 검색 성공 후 노트 열람률, RAG 출처 클릭률, AI 제안 수락률, import 후 재검색률, 유료 전환과 기능별 token 사용량을 함께 보겠습니다. 이벤트·feedback·usage 기록 기반은 있으나 North Star KPI의 기준선과 목표 수치는 저장소에서 확인되지 않으므로, 발표에서는 제안 지표와 실제 성과를 구분하겠습니다.
**근거:** [AI 기능 카탈로그], [LLMOps], [Commerce 사용량]

## 2. 아키텍처·서비스 경계

### 9. 전체 서비스 구조를 한 문장으로 설명한다면요?

**상태:** ✅ 구현됨
**모범 답변:** BrainX는 “Workspace 원장을 중심으로, User가 신원을 보장하고, Ingestion이 지식을 유입하며, Intelligence가 AI로 활용하고, Commerce가 권한과 비용을 관리하고, MCP와 Admin이 외부 agent와 운영을 연결하는 구조”입니다. Gateway는 외부 요청의 인증·라우팅을 담당하고 Discovery는 서비스 발견을 맡습니다. 이 설명은 기술 나열보다 데이터가 어느 도메인에서 생성·소유·소비되는지 보여 줍니다.
**근거:** [서비스 카탈로그], [OpenAPI SSOT], [Compose]

### 10. 공개 API와 서비스 간 API는 어떻게 구분하나요?

**상태:** ✅ 계약으로 정의됨
**모범 답변:** 브라우저와 외부 클라이언트는 기본적으로 `/api/v1/**` 공개 API를 사용하고, 서비스 간 동기 호출은 `/internal/v1/**` 계약과 `X-Service-Token` 경계로 분리합니다. Gateway는 외부 요청의 JWT를 검증해 사용자 식별 헤더를 주입하고, 내부 경로에는 별도 서비스 토큰을 요구합니다. 따라서 프론트가 다른 서비스 DB나 internal API를 우회해 호출하는 구조가 아니라, 계약된 진입점을 통해 도메인 책임을 지키게 됩니다.
**근거:** [SSOT README], [Gateway 게스트 인증]

### 11. 동기 REST와 Kafka 이벤트를 어떤 기준으로 나눴나요?

**상태:** ✅ 구현됨
**모범 답변:** entitlement 확인, 최신 Workspace snapshot 확인, 승인된 원본 변경처럼 즉시 결과가 필요한 일은 내부 REST를 사용합니다. 반면 노트 변경 전파, 검색 index 갱신, token usage 기록처럼 즉시 같은 트랜잭션으로 끝낼 필요가 없는 일은 Kafka 이벤트로 분리했습니다. AsyncAPI는 at-least-once 전달을 전제로 하므로 consumer는 idempotency와 DLQ를 함께 가져가야 합니다.
**근거:** [AsyncAPI 가이드], [교차 서비스 경계], [이벤트 체크포인트]

### 12. User-Service는 인증 외에 어떤 도메인을 책임지나요?

**상태:** ✅ 구현됨
**모범 답변:** User-Service는 local·소셜 로그인, 가입·온보딩, refresh session, 이메일 인증, 계정/보안 설정, 동의·알림과 사용자 문의를 맡습니다. 가입 또는 OAuth 온보딩 뒤 기본 Workspace provisioning을 시도해 사용자가 바로 기록을 시작하게 연결합니다. MCP OAuth authorization server 역할도 User-Service가 맡지만, 실제 MCP 도구 실행과 API client 관리는 MCP-Service의 책임으로 분리했습니다.
**근거:** [User auth controller], [User service], [OpenAPI SSOT]

### 13. Workspace-Service를 단순 노트 CRUD로 보면 부족한 이유는 무엇인가요?

**상태:** ✅ 구현됨
**모범 답변:** Workspace는 노트 본문뿐 아니라 폴더 트리, 태그, 위키 링크, backlinks, 즐겨찾기, 공유 링크, graph layout, draft와 version conflict까지 관리하는 지식 원장입니다. 내용 변경에는 `baseVersion`을 요구해 충돌을 409로 돌려주고, guest draft와 member 영구 데이터를 분리합니다. 따라서 Workspace는 화면 하나의 backend가 아니라 모든 채널이 공유하는 사용자 지식의 일관성 경계입니다.
**근거:** [Workspace README], [Workspace 정책], [Workspace API 매핑]

### 14. Ingestion-Service는 어떤 비즈니스 문제를 해결하나요?

**상태:** ✅ 구현됨
**모범 답변:** 사용자가 이미 가진 자료를 새로 복사해 붙여 넣지 않아도 BrainX 지식으로 옮길 수 있게 하는 것이 Ingestion의 역할입니다. 파일 업로드와 변환, Notion OAuth/import, Obsidian·파일 import, 웹 확장 캡처, export와 publish job을 담당하고, 결과 노트 생성은 Workspace 내부 command로 요청합니다. 즉 Ingestion은 콘텐츠의 입구와 변환을 맡되 원본 노트의 소유권을 가져가지 않습니다.
**근거:** [Ingestion controller], [Ingestion import service], [Chrome Extension API]

### 15. Intelligence-Service를 원본 서비스와 분리한 이유는 무엇인가요?

**상태:** ✅ 구현됨
**모범 답변:** AI 검색과 생성은 원본 CRUD와 다른 저장·조회 패턴을 필요로 합니다. Intelligence는 Workspace 이벤트와 snapshot을 받아 chunk manifest, embedding, Qdrant index, 요약·대화·분석 결과를 만들고, 원본 변경은 승인된 Workspace command로만 처리합니다. 이렇게 하면 model이나 vector 기술을 바꿔도 Workspace의 원본 도메인을 직접 바꾸지 않고, 색인 지연은 projection 상태로 별도 관리할 수 있습니다.
**근거:** [Intelligence 데이터 소유권], [노트 chunk·index], [Qdrant·Voyage]

### 16. Commerce-Service는 결제 화면만 담당하나요?

**상태:** ✅ 구현됨
**모범 답변:** Commerce는 plan, checkout session, Toss Payments confirm, subscription뿐 아니라 AI 기능을 실행해도 되는지 판단하는 entitlement와 token usage ledger를 맡습니다. 가격은 클라이언트가 아니라 서버 plan에서 정하고, 결제 confirm은 금액을 검증한 뒤 처리하며, AI 기능은 실행 전 Commerce에 권한을 확인합니다. 이 구조 덕분에 Intelligence가 결제 정책이나 잔여 credit을 직접 소유하지 않습니다.
**근거:** [Commerce README], [Commerce entitlement], [Commerce 사용량]

### 17. MCP-Service는 왜 필요한가요?

**상태:** ✅ 구현됨
**모범 답변:** MCP는 외부 AI client가 사용자의 BrainX 지식에 안전하게 접근하도록 하는 agent gateway입니다. 사용자는 scope가 있는 API client를 만들고, MCP는 `notes:read`, `notes:write`, `ai:search` 같은 권한을 확인해 semantic search, 노트 기반 질의, 단건 조회, 생성·삭제 도구를 제공합니다. wildcard scope는 허용하지 않으므로 외부 agent가 사용자 동의 없이 모든 기능을 수행하는 구조는 아닙니다.
**근거:** [MCP 노트 도구], [MCP 보안], [OpenAPI SSOT]

### 18. Admin-Service의 역할과 한계는 무엇인가요?

**상태:** ✅ 구현됨
**모범 답변:** Admin은 운영자가 사용자 상태·플랜·결제·환불·실패 결제·지원 티켓·token usage·서비스 모니터링을 한곳에서 처리하도록 합니다. 자체적으로 모든 원본 데이터를 복제하는 대신 User, Commerce, Workspace의 internal API를 호출해 dashboard와 운영 명령을 구성합니다. 따라서 Admin은 고객 운영의 조정 계층이며, 각 서비스의 기준 DB를 직접 수정하는 소유자가 아니라는 점을 강조하겠습니다.
**근거:** [Admin controllers], [Admin service], [OpenAPI SSOT]

### 19. Gateway와 Discovery는 비즈니스 도메인을 왜 갖지 않나요?

**상태:** ✅ 구현됨
**모범 답변:** Gateway는 JWT 검증, guest ID 발급, 헤더 위조 방지, route·circuit breaker를 담당해 외부 요청을 안전하게 각 서비스로 연결합니다. Discovery는 Eureka 기반 서비스 발견만 담당합니다. 이들이 노트나 결제 데이터를 소유하지 않게 하면, edge 정책이나 instance 위치가 바뀌어도 Workspace·Commerce 같은 비즈니스 원장의 책임은 흔들리지 않습니다.
**근거:** [Gateway 게스트 인증], [Gateway routes], [Compose]

### 20. 가입부터 첫 노트까지 어떤 흐름인가요?

### 21. 노트 편집 충돌과 데이터 유실을 어떻게 막나요?

**상태:** ✅ 구현됨
**모범 답변:** 노트 내용 변경에는 클라이언트가 알고 있는 `baseVersion`을 보내고, 서버 version과 다르면 `NOTE_VERSION_CONFLICT`로 충돌을 돌려줍니다. 멤버의 autosave는 Redis draft를 거쳐 영구 저장으로 flush하고, 게스트는 영구 노트 대신 Redis draft만 사용합니다. 이 설계는 동시 편집을 자동 병합했다고 주장하는 것이 아니라, 조용한 덮어쓰기를 피하고 사용자가 충돌을 인지하게 하는 방식입니다.
**근거:** [Workspace README], [Workspace 정책]

### 22. 파일·Notion·Obsidian·웹 페이지는 어떻게 노트가 되나요?

**상태:** ✅ 구현됨
**모범 답변:** Ingestion-Service는 파일과 zip, HTML·CSV·PDF·DOCX·PPTX 등의 변환, Notion OAuth 기반 import, Obsidian import와 웹 확장 캡처를 처리합니다. 변환 결과나 캡처 메타데이터는 Ingestion에 머무르지 않고 Workspace bulk create 또는 생성 command를 통해 사용자 노트가 됩니다. 이 흐름을 분리해 둔 덕분에 import가 늘어나도 노트의 folder·version·공유 규칙은 Workspace 정책을 그대로 적용합니다.
**근거:** [Ingestion controller], [Ingestion import service], [Chrome Extension API]

### 23. 공유와 graph는 어떤 사용자 가치를 주나요?

**상태:** ✅ 구현됨
**모범 답변:** 공유 링크는 노트를 외부에 전달하는 경로를 만들고, 그래프는 노트 간 위키 링크와 backlink를 시각화해 흩어진 지식의 관계를 보게 합니다. 이 둘은 Intelligence의 추정 결과가 아니라 Workspace가 소유한 원본 링크·공유 데이터 위에서 동작합니다. 따라서 AI 기능을 사용하지 않아도 사용자는 자신의 지식 구조를 관리할 수 있고, AI는 그 위에 연결 추천이나 분석을 더하는 역할입니다.
**근거:** [Workspace README], [Workspace API 매핑], [웹 라우트]

### 24. 외부 agent가 노트를 바꿀 수 있는데 안전한가요?

**상태:** ✅ 기본 보호 구현, ⚠️ 권한 모델 확장은 과제
**모범 답변:** MCP 도구는 인증된 API client의 userId와 scope를 먼저 확인하고, 읽기·검색·쓰기 권한을 분리합니다. 생성은 Workspace internal bulk-create를 사용하고 삭제는 사용자 범위의 internal delete 명령을 사용하므로 도구가 DB를 직접 만지지 않습니다. 다만 팀 공유 Workspace나 세분화된 resource permission을 도입한다면 현재의 user 중심 scope만으로는 부족할 수 있어, resource-level authorization을 별도로 설계해야 합니다.
**근거:** [MCP 노트 도구], [MCP Workspace gateway], [MCP 보안]

### 25. RAG 대화는 어떤 흐름으로 사용자 가치가 되나요?

**상태:** ✅ 구현됨

**근거:** [RAG 채팅], [AI 기능 카탈로그]

### 26. AI가 노트를 마음대로 수정하지 않게 한 이유는 무엇인가요?

**상태:** ✅ 구현됨

## 4. AI·RAG·데이터 품질

### 27. 왜 pgvector가 아니라 Qdrant와 Voyage를 사용하나요?

**상태:** ✅ 구현됨
**모범 답변:** 현재 vector 경로는 pgvector나 Spring AI VectorStore convenience가 아니라, AiEmbeddingPort를 통한 Voyage embedding과 Qdrant Java client/gRPC를 직접 사용합니다. 이 구조는 document·query embedding 사용량, chunk별 부분 갱신, userId·documentGroupId filter와 비용 추정을 세밀하게 통제하기 위한 선택입니다. chat adapter는 Spring AI를 활용하되, vector 경로는 provider 교체 가능성을 위해 별도 port로 분리했습니다.
**근거:** [Qdrant·Voyage], [노트 chunk·index]

### 28. 노트 수정 후 검색 품질과 비용을 어떻게 함께 관리하나요?

**상태:** ✅ 구현됨
**모범 답변:** 기본 chunking은 최대 1,200자, overlap 150자, 노트당 최대 80개이며, manifest와 chunk text hash를 이용해 변화량이 작으면 바뀐 chunk만 embedding하고 Qdrant를 upsert합니다. title·chunker 규칙 변경, manifest 부재, 큰 변경 비율에서는 full replace를 수행합니다. 즉 검색 최신성을 확보하면서도 작은 수정마다 노트 전체를 재embedding하는 비용을 피하는 구조입니다.
**근거:** [노트 chunk·index], [Qdrant·Voyage]

### 29. 이벤트 중복·역순·색인 실패는 어떻게 다루나요?

**상태:** ✅ 구현됨
**모범 답변:** Kafka는 at-least-once라 같은 이벤트가 다시 오거나 순서가 달라질 수 있으므로 소비 기록과 도메인별 idempotency 판단을 둡니다. version 정보가 불완전한 변경 이벤트는 payload를 맹신하지 않고 최신 Workspace snapshot을 다시 읽어 projection을 맞춥니다. 색인 상태는 NOT_INDEXED, PROVISIONAL, STALE, INDEXED, FAILED, REMOVED로 관리하고 ScheduledNoteReindexWorker가 실패 대상을 재시도합니다.
**근거:** [이벤트 체크포인트], [노트 chunk·index], [AsyncAPI 가이드]

### 30. RAG 환각과 최신 정보 문제를 어떻게 줄이나요?

**상태:** ✅ 완화 장치 구현, ⚠️ 완전 제거 아님
**모범 답변:** RAG는 route별 prompt로 검색된 노트 근거 안에서 답하게 하고, 관련 context가 부족할 때 근거 없는 답을 만들지 않는 흐름을 둡니다. 최신 외부 정보는 router가 필요하다고 판단하고 OpenAI provider·key가 설정된 경우에만 web search를 사용하며, 기본 provider는 none이라 항상 외부 검색하는 것은 아닙니다. 출처를 사용자에게 보여 주고, 검색/생성 품질이 완전하다고 과장하지 않는 것이 핵심입니다.
**근거:** [RAG 채팅], [외부 검색], [AI 기능 카탈로그]

### 31. 인라인 작성 보조와 문체 개인화는 어떻게 다르나요?

**상태:** ✅ 구현됨
**모범 답변:** 인라인 작성 보조는 선택한 텍스트와 앞뒤 문맥으로 요약·재작성·이어쓰기·번역·초안을 만드는 기능입니다. 문체 프로필은 대화 설명에는 `conversationTone`, 저장할 결과물에는 `writingStyle`을 적용해 사용자가 실제로 재사용할 만한 결과를 만들려는 장치입니다. 다만 안전성, 사실성, 출력 형식과 사용자의 명시 지시는 개인 문체보다 우선합니다.
**근거:** [AI 기능 카탈로그], [StyleProfile 방향], [Inline Assist SSE]

### 32. 클러스터와 인사이트 API가 202인데 실제 background 처리인가요?

**상태:** ⚠️ API 표면과 v1 구현의 차이
**모범 답변:** v1은 job ID와 상태 조회를 제공하기 위해 202 Accepted를 반환하지만, 현재 POST 요청 안에서 note card 조회, entitlement 확인, LLM 호출, 결과 저장까지 sync-complete로 수행합니다. 이를 진짜 비동기 worker라고 부르면 안 됩니다. 대량 분석으로 확장할 때는 요청은 idempotent job 생성만 하고 worker가 실행·재시도·취소·진행 상태를 관리하도록 분리해야 합니다.
**근거:** [지식 구조 분석], [인사이트 리포트]

### 33. AI 품질은 어떻게 관측하고 개선하나요?

**상태:** ✅ LLMOps 기반 구현, ⚠️ 운영 UI는 후속
**모범 답변:** provider 호출에는 llmRunId, prompt key/version, latency, token·cost estimate와 마스킹한 preview를 기록하고, LIKE·DISLIKE feedback과 eval scenario를 연결할 수 있습니다. 이 정보는 billing ledger가 아니라 모델·prompt 품질과 비용을 진단하기 위한 관측 데이터입니다. v1 eval은 요청 안에서 동기 실행되고 전용 운영 UI나 background eval worker는 아직 없으므로, LLMOps 기반과 운영 체계 완성을 구분해 설명하겠습니다.
**근거:** [LLMOps], [LLM 품질 평가], [Commerce 사용량]

## 5. 보안·운영·신뢰성

### 34. API 인증과 내부 서비스 인증은 어떻게 다른가요?

**상태:** ✅ 구현됨
**모범 답변:** 외부 요청은 Gateway 또는 각 서비스에서 bearer JWT를 검증해 사용자 identity를 전달하고, `/internal/v1/**` 서비스 호출은 `X-Service-Token`으로 분리합니다. Gateway는 외부에서 보낸 `X-User-*`, `X-Guest-Id` 헤더를 먼저 제거해 위조를 막고, 검증한 identity만 다시 주입합니다. local 개발 profile의 인증 완화는 테스트 편의를 위한 것이므로 운영 보안으로 설명하지 않습니다.
**근거:** [Gateway 게스트 인증], [SSOT README], [User auth controller]

### 35. 다른 사용자의 노트가 검색되거나 수정되는 위험은 어떻게 막나요?

**상태:** ✅ 구현됨
**모범 답변:** 검색 projection과 Qdrant filter의 핵심 범위는 userId와 documentGroupId이며, Workspace 전체 검색만 의도적으로 userId 전체 범위를 사용합니다. Agent mutation은 projection의 소유권뿐 아니라 최신 Workspace snapshot의 group·version을 다시 확인합니다. MCP도 현재 API client의 userId와 scope를 기준으로 downstream 호출을 만들기 때문에, 클라이언트가 보낸 임의 userId를 신뢰해 노트를 찾는 방식이 아닙니다.
**근거:** [Qdrant·Voyage], [교차 서비스 경계], [MCP 노트 도구]

### 36. 프롬프트 인젝션과 위험한 AI action에 어떻게 대응하나요?

**상태:** ✅ 기본 안전장치 구현, ⚠️ 전용 탐지기는 미구현
**모범 답변:** Agent action은 allow-list한 CREATE_NOTE와 APPEND_NOTE_CONTENT만 저장하고, 실제 변경 전 사용자 승인을 요구합니다. 승인 뒤에도 소유권과 version을 확인해 Workspace command만 호출합니다. 이는 blast radius를 줄이는 안전장치이지 전용 prompt-injection 탐지·차단 체계나 공유 Workspace의 세밀한 권한 모델까지 완성했다는 뜻은 아닙니다.
**근거:** [AI 기능 카탈로그], [교차 서비스 경계]

### 37. LLM 로그와 민감 데이터는 어떻게 다루나요?

**상태:** ✅ 구현됨, ⚠️ 완전 PII/DLP는 아님
**모범 답변:** LLMOps는 full prompt와 full output을 기본 저장하지 않고 길이가 제한된 JSON preview를 남기며, 알려진 credential pattern을 마스킹합니다. 하지만 기본 최대 2,000자의 사용자 입력·출력 일부가 preview에 남을 수 있으므로 이것을 완전한 개인정보 DLP라고 부를 수는 없습니다. 운영에서는 보존 기간, 접근 권한, 추가 PII 탐지와 provider key의 secret 관리를 함께 보강해야 합니다.
**근거:** [LLMOps], [Intelligence README]

### 38. 사용자가 탈퇴하면 모든 데이터가 삭제되나요?

**상태:** ⚠️ 현재 한계
**모범 답변:** Intelligence는 UserDeletionRequested 이벤트를 받아 삭제 예정 시각과 사유를 idempotent하게 projection으로 기록합니다. 그러나 AI 설정, vector index, summary cache, chat·agent 기록, cluster·insight 결과를 일괄 purge하는 worker는 아직 구현되지 않았습니다. 따라서 삭제 요청 수신 기반은 갖췄지만 완전한 AI 데이터 삭제는 출시 전 우선순위가 높은 후속 과제라고 답하는 것이 정확합니다.
**근거:** [삭제 handler], [이벤트 체크포인트], [Intelligence 데이터 소유권]

### 39. 이벤트 실패와 DLQ는 어느 수준까지 준비됐나요?

**상태:** ✅ DLQ 전송 코드, ⚠️ 운영화 미완성
**모범 답변:** retryable 오류는 정해진 횟수만큼 재시도한 뒤 같은 partition 기준의 `.dlq` topic으로 보내고, invalid payload 같은 non-retryable 오류는 즉시 DLQ로 보냅니다. DLQ publish 실패를 정상 recover로 처리하지 않아 원본 offset이 조용히 사라지는 것도 막습니다. 다만 DLQ topic 생성, alert, 원인 분류와 재처리 runbook은 배포 전 별도로 준비해야 실제 운영 복구가 가능합니다.
**근거:** [이벤트 체크포인트], [AsyncAPI 가이드], [AWS 감사]

### 40. 현재 운영 토폴로지와 가장 큰 위험은 무엇인가요?

**상태:** ⚠️ 단일 host 운영
**모범 답변:** 확인된 운영 기준은 단일 EC2에서 Docker Compose로 backend 서비스와 Kafka, Redis, Neo4j, Qdrant, 관측 도구를 함께 실행하는 구조입니다. 비용과 초기 운영 복잡도는 낮지만 host 장애가 여러 상태 저장 의존성과 관측까지 동시에 멈추게 하는 single point of failure입니다. Kubernetes나 Multi-AZ가 이미 운영 중이라고 주장하지 않고, 현재 구조와 이관 준비를 분리해 설명하겠습니다.
**근거:** [AWS 런타임], [Compose], [AWS 감사]

### 41. 무엇을 관측하고, 무엇이 부족한가요?

**상태:** ✅ 일부 관측 확인, ⚠️ 외부 관측성 부족
**모범 답변:** Prometheus·Grafana·Loki 구성과 application actuator metric 경로는 마련되어 있고, 감사 시점에는 Prometheus target, Qdrant, Kafka lag, Intelligence restart 상태를 점검했습니다. 하지만 모니터링 스택도 같은 EC2에 있어 host 장애 때 증거를 함께 잃을 수 있고, CloudWatch alarm·application log group·RDS 관측성은 부족하다는 지적이 있습니다. 다음 단계에서는 DB·host·DLQ·의존성별 alarm과 외부 log 보관을 우선 보강해야 합니다.
**근거:** [AWS 런타임], [AWS 감사]

### 42. AWS root principal과 RDS 위험에는 어떻게 답하겠나요?

**상태:** ⚠️ 즉시 보완 필요
**모범 답변:** 감사에서 AWS CLI session이 root principal이었던 점은 일상 운영으로 허용하면 안 되는 가장 높은 위험입니다. 배포와 운영은 최소 권한 IAM role 또는 Identity Center session으로 옮기고, root는 MFA 기반의 비상 접근만 허용하며 root access key를 제거해야 합니다. 동시에 감사에서 확인된 RDS Single-AZ, storage encryption·deletion protection 비활성, 짧은 backup retention, alarm 부재도 복원력 계획으로 명확히 관리하겠습니다.
**근거:** [AWS 감사]

## 6. 비용·결제·사업 운영

### 43. AI 비용은 어디서 발생하고 어느 서비스가 기록하나요?

**상태:** ✅ 구현됨
**모범 답변:** 비용은 chat, router, external search, summary, 연결·분석 기능의 LLM 호출과 document/query embedding에서 발생합니다. Intelligence는 provider usage 또는 streaming 추정치로 TokenUsageRecord를 만들고 이벤트를 발행하며, Commerce가 이를 받아 credit 정책과 billing ledger를 소유합니다. 따라서 LLMOps run log는 품질·관측용이고, vendor 원가 estimate는 청구 원장과 다르다는 구분을 분명히 해야 합니다.
**근거:** [Commerce 사용량], [LLMOps], [교차 서비스 경계]

### 44. 기능 entitlement와 모델별 요금제를 어디까지 구현했나요?

**상태:** ✅ 기능 entitlement, ⚠️ 모델별 연동은 미구현
**모범 답변:** AI 기능 호출 전에는 Intelligence가 Commerce entitlement check를 실제로 호출하고, 응답 오류나 불완전한 응답은 fail-closed로 AI 실행을 막습니다. 반면 모델 목록의 enabled 여부를 Commerce policy로 해석하는 availability adapter는 현재 `none` 정책을 반환하는 placeholder입니다. 즉 기능 사용권 방어는 구현됐지만 모델별 플랜 노출·제한까지 완성됐다고 말하면 안 됩니다.
**근거:** [Commerce entitlement], [Entitlement adapter], [Model availability adapter]

### 45. 결제 플로우는 실제로 어떻게 동작하고 무엇이 남았나요?

**상태:** ✅ Toss confirm 구현, ⚠️ 운영 정산 보강 필요
**모범 답변:** Commerce는 서버 plan 가격을 기준으로 checkout session을 만들고, confirm 전에 금액을 검증한 뒤 Toss Payments SDK/API를 호출합니다. checkout은 pending에서 succeeded·failed·expired 상태로 관리하고 idempotent confirm을 고려합니다. 하지만 webhook 기반 reconciliation은 README에서 미구현으로 명시되어 있고, 개발용 공개 접근·테스트 가격 등도 남아 있어 production 결제 완성도와 구분해야 합니다.
**근거:** [Commerce README], [Commerce controller]

### 46. 월간 COGS와 예산 초과는 어떻게 관리하나요?

**상태:** 🗺️ 후속 과제
**모범 답변:** 현재 구현 범위는 provider token usage와 vendor 단가 기반 estimate, 기능별 usage event, Commerce의 credit/usage 조회입니다. 월간 트래픽 가정과 기능별 호출률, tool-call 비용, credit 정책을 합친 COGS forecast나 예산 alert dashboard는 확인되지 않았습니다. 다음 단계에서는 사용자·기능별 비용 상한, budget alert, cohort별 매출·원가 분석을 Commerce와 Admin 경계에서 설계하겠습니다.
**근거:** [Commerce 사용량], [AI 모델 가격], [Admin controllers]

## 7. 로드맵·협업·개인 기여

### 47. 출시 전 우선순위는 어떻게 정하겠나요?

**상태:** 🗺️ 후속 과제
**모범 답변:** 새 AI 기능보다 먼저 데이터 유실·권한·장애 감지 위험을 낮추겠습니다. 우선 root 일상 사용 중단과 IAM 전환, RDS 보호·백업·암호화·alarm, DLQ topic·alert·runbook, 외부 관측성, dependency readiness, 사용자 AI 데이터 purge를 순서대로 보강하겠습니다. 그 다음 cluster·insight의 실제 worker 전환, 모델별 entitlement, provider health 기반 failover와 desktop local-first sync를 확장하는 것이 안전합니다.
**근거:** [AWS 감사], [이벤트 체크포인트], [Electron README], [삭제 handler]

### 48. 본인의 기여는 어떻게 설명하면 좋나요?

**상태:** 발표자 입력 필요
**모범 답변 템플릿:** “저는 **[맡은 도메인/서비스]**에서 **[사용자 또는 운영 문제]**를 해결하기 위해 **[원본 소유권·계약·보안·비용·성능 중 실제 설계 결정]**을 했습니다. 결과는 **[API·코드·테스트·운영 문서]**로 검증했고, 남은 한계인 **[후속 과제]**도 분리해 관리했습니다. 팀과는 **[OpenAPI/AsyncAPI 계약 조율, 리뷰, 배포, 문서화]**로 연결했습니다.”
**주의:** 다른 팀원의 결과나 계획 문서를 본인 기여처럼 말하지 말고, 직접 설명하고 증명할 수 있는 범위만 채운다.

- 개인 기여는 본인이 실제로 설명·증명할 수 있는 내용만 채웠는가?
